# Milestone v1.6: Security and Abuse Hardening

**Status:** ✅ SHIPPED 2026-03-11
**Phases:** 28-30
**Total Plans:** 6

## Overview

Milestone focado em endurecer a API contra abuso automatizado e regressões de segurança, com throttling em Rack, postura explícita de produção e gates estáticos obrigatórios.

## Phases

### Phase 28: Rack Abuse Boundary

**Goal**: Introduzir throttling em Rack para auth, tokens e endpoints de alto risco com contrato 429 consistente.
**Depends on**: Phase 27
**Plans**: 2 plans

Plans:

- [x] 28-01: boundary inicial em Rack para signup/login/refresh user/admin, contrato `429` uniforme e preservação do `/up`
- [x] 28-02: extensão do throttle para `cart/checkout`, mutações de carrinho e ações admin sensíveis, com fechamento de regressão

**Details:**
- Requirements: `ABUSE-01`, `ABUSE-02`, `ABUSE-03`
- Success criteria:
  1. Login/signup/refresh user e admin recebem throttles distintos e adequados ao risco.
  2. Endpoints write/high-cost críticos recebem throttles por ator autenticado com fallback por IP.
  3. Resposta de bloqueio é `429` estável e não revela se credencial ou recurso existe.
  4. Eventos de throttle ficam rastreáveis em logs/telemetria mínima para suporte operacional.

### Phase 29: Production Security Posture

**Goal**: Endurecer configuração de produção para SSL, hosts, proxy e CORS sem quebrar Railway ou healthchecks.
**Depends on**: Phase 28
**Plans**: 2 plans

Plans:

- [x] 29-01: baseline explícito de produção para SSL, host authorization e validação de envs críticos
- [x] 29-02: CORS orientado por ambiente, regressão de origem permitida/bloqueada e fechamento formal de `SEC-01` e `SEC-02`

**Details:**
- Requirements: `SEC-01`, `SEC-02`
- Success criteria:
  1. Produção define política clara para SSL, proxy confiável e headers/hosts aceitos.
  2. CORS deixa de depender de origem localhost hardcoded e passa a seguir configuração por ambiente.
  3. Healthcheck `/up` e domínio Railway seguem funcionais após o hardening.
  4. Configuração falha de forma explícita quando variáveis críticas de segurança estiverem ausentes.

### Phase 30: Static Security Gates and Regression Net

**Goal**: Consolidar validações estáticas de segurança e testes de regressão para throttling e guardrails críticos.
**Depends on**: Phase 29
**Plans**: 2 plans

Plans:

- [x] 30-01: gate único fail-closed para `bundler-audit` + `brakeman` e integração ao CI do repositório
- [x] 30-02: comando explícito de regressão de segurança/hardening e fechamento formal de `SEC-03` e `SEC-04`

**Details:**
- Requirements: `SEC-03`, `SEC-04`
- Success criteria:
  1. Existe um comando/fluxo único para rodar `bundler-audit` e `brakeman` em modo obrigatório.
  2. O gate de segurança é integrado ao fluxo padrão de CI do projeto.
  3. Testes cobrem o caminho feliz e o bloqueio dos throttles principais.
  4. Alterações futuras em rotas/configuração de segurança quebram testes ou checks quando regressivas.

## Milestone Summary

**Key Decisions:**

- Aplicar rate limiting no Rack antes dos controllers para bloquear bursts cedo e reduzir custo por request.
- Manter `/up` fora do boundary de abuso e também fora de partes críticas do hardening de produção para não quebrar probes do Railway.
- Centralizar validação estática em `bin/security` e consolidar regressão crítica em `bin/security-regression`.
- Executar a suíte completa em modo serial no host atual quando o `pg` em paralelo se mostra instável.

**Issues Resolved:**

- Auth, cart, checkout e mutações admin sensíveis ganharam throttles previsíveis com contrato `429` único.
- Produção passou a falhar explicitamente quando envs de segurança estão ausentes, em vez de depender de defaults frágeis.
- O projeto passou a ter enforcement local e remoto para `bundler-audit` e `brakeman`.
- O lockfile foi atualizado para remover uma vulnerabilidade real detectada pelo gate estático.

**Issues Deferred:**

- Smoke externo do domínio público do Railway após o hardening de produção ainda precisa ser executado.
- Execução observada do workflow remoto no provedor Git ainda não foi confirmada.

**Technical Debt Incurred:**

- `PROJECT.md` exigiu alinhamento pós-milestone porque ainda refletia v1.6 como ativo.
- A operação local continua dependendo de suíte serial para evitar instabilidade do `pg` em paralelo no host atual.
