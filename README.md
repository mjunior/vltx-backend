## Overview

**EN**

This repository is a deliberate VIBECODE experiment. The entire codebase, prompt trail, planning artifacts, and delivery flow were authored by AI with assistance from the [get-shit-done](https://github.com/glittercowboy/get-shit-done) workflow/skill. No line was manually edited.

All phases are available for inspection under `.planning/`, including plans, validation files, and execution artifacts that expose how the prompts evolved and how each delivery step was structured.

The core question is not whether AI can ship software. It is whether security failures in AI-built systems emerge primarily from:

1. operator inexperience and incomplete prompting,
2. missing security intent in the requested scope,
3. or AI negligence even when secure behavior should be inferred.

This project turns that question into something inspectable: a real Rails API, a visible repository history, explicit security gates, and a codebase that can be reviewed without the usual "someone cleaned it up later" ambiguity.

**PT-BR**

Este repositório existe para validar uma hipótese simples: quando falhas de seguranca aparecem em um sistema gerado por IA, a causa principal esta na falta de experiencia de quem opera o fluxo e deixa de pedir requisitos importantes, ou na propria IA, que negligencia cuidados essenciais mesmo quando o contexto aponta nessa direcao.

Todo o codigo, prompts, estrutura de planejamento e artefatos deste repositorio foram escritos por IA com apoio do workflow/skill [get-shit-done](https://github.com/glittercowboy/get-shit-done). Nenhuma linha foi alterada manualmente. O objetivo nao e fingir perfeicao; o objetivo e deixar o experimento audivel, reproduzivel e honesto.

Todas as phases, plans, validacoes e rastros de execucao ficam disponiveis em `.planning/` para analise do processo e dos prompts que conduziram cada entrega.

## What This Project Is

`marketplace_backend/` contains a Rails 8 API for a marketplace domain with:

- user authentication with access and refresh JWT flows
- admin authentication and admin-only operational endpoints
- public product listing and product detail endpoints
- seller product management
- cart and checkout flows
- order lifecycle transitions, cancellation, delivery, and contestation
- wallet and seller finance endpoints
- transactional email support for password reset flows
- security-oriented checks wired into the local and CI workflow

## Why This README Exists

Most READMEs describe a product. This one also documents an authorship model.

If you are auditing this repository, the important claim is straightforward:

- 100% of the code was generated by AI
- 100% of the prompts and repository artifacts were generated by AI
- 0% of the code was manually rewritten after generation

That makes this repository useful beyond the marketplace use case. It is also a test bench for:

- prompt quality versus implementation quality
- implied versus explicit security requirements
- how much secure behavior AI adds by default
- how much risk remains when humans do not ask the right questions

## Design Principle

This repository is intentionally transparent, not mythologized.

It does **not** claim:

- that AI-written code is automatically secure
- that passing static checks means the system is production-safe
- that a generated codebase is inherently lower or higher quality than hand-written code

It **does** claim:

- the authorship model is explicit
- the experiment can be reviewed in public
- the security posture can be challenged with actual code, tests, and configuration

## Security Framing

Security is part of the experiment, so the repository already includes visible hardening signals such as:

- `bin/security` for static fail-closed checks
- `bundler-audit` for vulnerable dependency detection
- `brakeman` for Rails-focused security analysis
- `bin/security-regression` for targeted hardening coverage
- `rack-attack` middleware for abuse throttling
- production security initialization checks
- focused integration tests around auth, CORS, healthcheck behavior, and admin boundaries

That matters because this repo is not trying to answer "can AI scaffold a CRUD app?"

It is trying to answer the harder question:

**What does AI secure on its own, and what only becomes secure when a human explicitly demands it?**

ADMIN_PASSWORD='change-this-password' \

## How To Read This Repository

If you are here as an engineer, reviewer, or security auditor, evaluate it in this order:

1. Read the auth and operational docs.
2. Inspect the security scripts and initializers.
3. Review integration tests around trust boundaries.
4. Verify whether protections were inferred by the AI or only added after specific prompting.

That sequence is the whole point of the experiment.

## Important Limitation

This repository demonstrates an AI-authored system. It is not a universal statement about all AI-generated systems.

Any conclusion you draw should stay narrow:

- about this repository,
- about this operator workflow,
- about these prompts,
- and about the security posture observable in this codebase.

## Final Statement

This is not "AI-assisted" code.

This is a repository where the code, the prompts, the planning trail, and the implementation artifacts were all produced by AI so the failures, strengths, omissions, and security posture can be judged without manual cleanup hiding the result.
